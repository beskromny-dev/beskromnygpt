## 1. Видение продукта

БескромныйAI — персональная голосовая модель Дмитрия Бескромного, упакованная в публичный продукт. Система знает каждый пост, каждый подкаст, каждое интервью и каждую транзакцию — и использует этот контекст для полуавтоматизации всего объёма коммуникационных задач: посты, комментарии, статьи, колонки, тезисы для выступлений.

Это одновременно три вещи:

**Для себя** — операционная система личного бренда. Нулевое трение, нулевой ввод контекста, 80% механической работы убрано.

**Для сцены** — живой публичный кейс. Продукт полностью построен на vibe-кодинге, без найма разработчиков. История которую можно показывать на ПМЭФ, ВЭФ, ЦИПР и продавать через COGNITA.

**Для рынка** — тиражируемая система для публичных людей и брендов. Первый прецедент в России.

> **Амбиция:** показать что медиа-предприниматель с растущими навыками vibe-кодинга может за месяц построить AI-систему которую агентства продают за миллионы.

## 2. Целеполагание и use cases

Это главный раздел. Вся архитектура — под него.

### 2.1 Первичные задачи — ежедневно, через Telegram-бот

**Rewrite чужого материала**
Переслать пост из любого канала → получить адаптацию в голосе Димы → опубликовать. Сейчас это делается вручную через ChatGPT без понимания голоса. Бот знает голос полностью.

**Комментарии и реакции**
Вставить текст чужого поста → получить 2–3 варианта ответа в тональности Димы → выбрать, поправить 10%, опубликовать. Время: < 3 минут.

**Посты для канала**
Дать инфоповод или тезис → система проверяет нет ли повтора в архиве → генерирует по пятичастной структуре с вариантами заголовков → Дима правит финал.

**Research + синтез**
Тема → web search → синтез → наложение на собственные мысли из архива → готовый материал с источниками.

**Статьи для Forbes / РБК**
Тема + издание → система вытаскивает все релевантные мысли из подкастов, интервью, постов → черновик в публицистическом регистре.

### 2.2 Вторичные задачи — еженедельно, через веб-интерфейс

- Аналитика канала: что заходит, что нет, корреляции тем и ER
- Финансовая динамика: выручка по источникам, связь роста аудитории и ставок
- Стратегический анализ: белые пятна в контенте, точки роста

### 2.3 Третичные задачи — ситуативно

- Тезисы для выступлений: тема конференции → структура + speaker notes
- Разработка новых продуктов на основе анализа болей аудитории
- Подготовка нативных интеграций для клиентов канала

## 3. Интерфейсная архитектура

Три инструмента. Не больше.

### 3.1 Telegram-бот — ежедневная работа

Основной рабочий интерфейс. Дима уже в Telegram постоянно — нулевое трение.

**Команды:**
- `/post [тема]` → пост в структуре канала, 3 варианта заголовка
- `/rewrite` + текст → адаптация в голосе Димы
- `/comment` + текст → ответ на комментарий
- `/research [тема]` → web search + синтез + мысли из архива
- `/article [тема] [издание]` → черновик статьи
- `/check [тезис]` → «ты уже писал об этом?»
- `/graph [период]` → график аналитики или финансов картинкой

**Фидбек прямо в боте:**
После каждой генерации — три кнопки: ✓ Использовал / ✎ Отредактировал / ✗ Не подошло. Пишется в SQLite автоматически.

**Saved Messages как источник:**
Через Telethon читается всё сохранённое в избранном — тексты, файлы, ссылки — и загружается в RAG как дополнительный пласт контекста.

### 3.2 Next.js веб-приложение — витрина и аналитика

Это то что показывается на сцене и продаётся через COGNITA. Тёмный минималистичный дизайн. Деплой на Vercel — живой URL, открывается с любого устройства.

**Три раздела:**

*Чат с голосом* — интерфейс как ChatGPT, но это Дима. Любой человек может написать тему и получить текст в голосе Бескромного. Демонстрационный режим для сцены.

*Аналитика канала* — живые данные LiveDune. Динамика подписчиков, ER по рубрикам, тепловая карта публикаций, топ постов. Красивые графики, тёмная тема, выглядит как продукт.

*Экономика бренда* — данные Т Банк API + Google Sheets. Динамика выручки, разбивка по источникам, корреляция аудитории и дохода.

**Почему Next.js, не Streamlit:**
Claude генерирует Next.js компоненты с Tailwind идеально — это его сильнейшая сторона. Vercel деплоит за 30 секунд из GitHub. Результат выглядит как стартап, а не как аналитическая утилита. Для публичного продукта и выступлений — принципиальная разница.

### 3.3 Google Docs — структурные документы

Уже подключён через MCP. Claude читает напрямую.

**Что хранится:**
- Voice Profile — главный документ системы (см. Блок 0)
- Мастер-документ: структура тем, белые пятна, стратегия контента
- Калибровочные примеры: топ-20 постов по ER как эталон голоса

**Obsidian — опционально, Фаза 2+.** Граф связей между темами — красиво и полезно, но не на старте. Сначала Google Docs, потом при необходимости переедет в Obsidian.

## 4. Блоки системы

### Блок 0 — Voice Profile (приоритет №1)

Явный артефакт — структурированный документ в Google Docs, который описывает голос Димы как систему. Создаётся до написания первой строки кода.

**Что входит:**

*Лингвистические паттерны*
- Запрещённые конструкции: «Мы привыкли думать», клише инфостиля, GPT-обороты
- Обязательные ограничения: без emoji, без восклицательных знаков, без em-dash как списков
- Любимые приёмы: короткий парадоксальный заголовок, наблюдение или ирония как финал, CTA ротацией

*Структурные паттерны по регистрам*
- Канал: байтовый заголовок → проблематизация → тезисы → панчлайн → CTA
- Публицистика: три блока ВИДЕТЬ / СВЯЗЫВАТЬ / ДЕЙСТВОВАТЬ
- Выступление: speaker notes, плотность аргументов, работа с аудиторией

*Тематический граф*
- Ключевые темы: внимание, доверие, AI, медиа, личный бренд, коммуникации
- Темы которые Дима трактует специфически или избегает

*Калибровочные примеры*
- Топ-20 постов по ER — эталон голоса канала
- Лучшие фрагменты интервью — эталон разговорного регистра
- Статьи Forbes/РБК — эталон публицистического регистра

**Создаётся:** полуавтоматически — Python анализирует корпус, предлагает паттерны, Дима верифицирует. Обновляется раз в квартал + через фидбек-петлю.

> Voice Profile — это разница между «похожий стиль» и «именно Дима».

---

### Блок A — Контентная память (RAG-ядро)

Весь текстовый архив в ChromaDB с метаданными и регистровым разделением.

**Регистры:**

| Регистр | Источник | Особенность |
|---|---|---|
| Канал | Посты «Бескромный» | Письменный, редактированный, эталонный |
| Публицистика | Forbes, РБК, Ведомости | Формальный, аргументированный |
| Разговорный | Подкасты, интервью | После постобработки |
| Выступления | ПМЭФ, ВЭФ, ЦИПР | Ораторский, структурированный |
| Saved Messages | Telegram избранное | Личный, референсный |
| Рабочий | Google Docs, брифы | Внутренний, не для генерации голоса |

**Метаданные каждого документа:**
- `register` — регистр из таблицы выше
- `date` — дата публикации
- `topic_tags` — тематические метки
- `er_score` — вовлечённость (для постов канала, из LiveDune)
- `type` — пост / статья / транскрипт / документ / saved

**Временное взвешивание:** посты последних 12 месяцев получают коэффициент ×1.5. Голос эволюционирует — свежее важнее.

**Постобработка транскриптов:** разговорная речь из PLAUD и Whisper проходит слой очистки перед загрузкой — убираются хезитации, нормализуется пунктуация. Без этого RAG засоряется артефактами устной речи.

**Источники и инструменты:**
- Telegram Bot API — архив канала, автообновление при новом посте
- Telethon — Saved Messages из избранного
- PLAUD — новые записи (лицензия есть)
- Whisper локально — архивные подкасты и интервью
- Google Docs MCP — документы (уже настроен)

---

### Блок B — Аналитическая панель

**Источники:** LiveDune API (основной), TGStat API (верификация)

**Метрики:**
- Динамика подписчиков — временной ряд
- ER по регистрам и рубрикам
- Тепловая карта: день × время публикации
- Корреляция: тема → ER
- Топ постов за любой период

**Связка с Блоком A:** ER из LiveDune автоматически пишется в метаданные поста в ChromaDB. Система знает что написано и как это приняла аудитория.

---

### Блок C — Экономическая панель

**Т Банк Бизнес API** — приоритетный источник
- Живые транзакции в реальном времени
- Автокатегоризация по клиентам и проектам: bQ Group, AI Influence, COGNITA, канал
- Ретроспектива без ручного ввода

**Google Sheets MCP** — дополнительный
- Pipeline сделок и прогноз
- Данные до подключения API

**Ключевая визуализация:**
- Корреляция: рост аудитории → рост ставки нативки
- Динамика по источникам дохода

---

### Блок D — Фидбек-петля

Без этого система не учится. Самое частое слепое место в персональных AI-инструментах.

**Механизм:**
- После каждой генерации в боте: ✓ / ✎ / ✗
- «Отредактировал» → сохраняется пара: [сгенерировано] → [финальный вариант]
- Раз в месяц: анализ паттернов ошибок → обновление Voice Profile

**Прогноз по правкам:**
- Месяц 1: 40–50% правок (норма, система калибруется)
- Месяц 2: 25–30%
- Месяц 3+: 10–15% (рабочий режим)

**Реализация:** SQLite локально. Лёгкая, приватная, не требует облака.

## 7. Период калибровки

Первые 4 недели после запуска — нормально получать 40–50% правок. Это не провал, это обучение.

**Что делать в этот период:**
- Активно нажимать кнопки фидбека после каждой генерации
- Фиксировать паттерны ошибок в отдельном разделе Google Docs
- Не менять запросы — менять Voice Profile

После первого анализа фидбека (конец месяца 1) — обновление Voice Profile. После этого качество растёт нелинейно.

## 8. Слепые места — зафиксированные решения

**Manus — не используется.** Python-скрипт делает то же структурированно и сразу в базу. Manus остаётся для одноразовых исследовательских задач вне системы.

**Streamlit — не используется.** Next.js закрывает всё что он делал, плюс выглядит как продукт. Для публичных выступлений — принципиальная разница.

**Obsidian — опционально, Фаза 3+.** Граф тем — полезно, но не на старте. Google Docs закрывает потребность на MVP и v1.0.

**Постобработка транскриптов — обязательна.** Разговорная речь без очистки засоряет голосовую модель.

**Временное взвешивание — да.** Коэффициент ×1.5 для постов последних 12 месяцев.

**Мультирегистровая модель — да.** Голос в канале и голос в Forbes — разные. С первого дня.

## 9. Риски и митигация

| Риск | Митигация |
|---|---|
| Voice Profile устаревает | Обновление раз в квартал + автоматически через фидбек |
| Транскрипты засоряют модель | Обязательная постобработка перед загрузкой |
| Регистровое смешение | Метаданные на каждый документ с момента загрузки |
| API лимиты Telegram | Загрузка через t.me/s/ + Bot API |
| Стоимость embeddings | Одноразово при загрузке, инкрементально после |
| Приватность финансовых данных | ChromaDB локально, никаких внешних сервисов |
| Деградация качества без фидбека | Фидбек-петля обязательна с первого дня |
| Масштабирование на COGNITA | Изоляция данных в архитектуре с самого начала |
